# Фронтенд и бэкенд разработка - Контрольная работа №4

В рамках данной контрольной работы были выполнены практические задания 19-24, посвященные реализации следующих функций:

--

## Практика 19: 

Посвящена работе с реляционными системами управления базами данных на примере СУБД PostgreSQL, в частности, созданию серверного приложения с подключением базы данных PostgreSQL для хранения информации о пользователях.

Сущность Пользователь включает следующий набор полей:

| Поле       | Тип данных | Описание                              |
| ---------- | ---------- | ------------------------------------- |
| ID         | Integer    | Уникальный идентификатор пользователя |
| first_name | Varchar    | Имя пользователя                      |
| last_name  | Varchar    | Фамилия пользователя                  |
| age        | Integer    | Возраст пользователя                  |
| created_at | Timestamp  | Время создания пользователя в unix    |
| updated_at | Timestamp  | Время обновления пользователя в unix  |

Список маршрутов включает следующий перечень энд-поинтов:

| Адрес          | Метод  | Описание                           |
| -------------- | ------ | ---------------------------------- |
| /api/users     | POST   | Создание нового пользователя       |
| /api/users     | GET    | Получение списка пользователей     |
| /api/users/:id | GET    | Получение конкретного пользователя |
| /api/users/:id | PATCH  | Обновление информации пользователя |
| /api/users/:id | DELETE | Удаление пользователя              |

## Практика 21:

Посвящена работе с Redis и применению Redis для кэширования. Было доработано приложение из практического занятия №11, где были:

- подключен Redis;
- создано middleware для чтения данных из кэша;
- создана функция сохранения данных в кэш;
- добавлена очистка кэша при изменении пользователей и товаров.

Кэширование было добавлено для следующих маршрутов:

| Маршрут           | Метод | Время кэша | Описание маршрута             |
| ----------------- | ----- | ---------- | ----------------------------- |
| /api/users        | GET   | 1 минута   | Получить список пользователей |
| /api/users/:id    | GET   | 1 минута   | Получить пользователя по id   |
| /api/products     | GET   | 10 минут   | Получить список товаров       |
| /api/products/:id | GET   | 10 минут   | Получить товар по id          |

Запуск Redis происходил с помощью команды docker run, для выполнения которой был установлен docker desktop для windows.

## Практика 22:

Посвящена балансировке нагрузки, настройке Nginx и HAProxy в роли балансировщиков, а также базовые подходы к развертыванию сбалансированных систем с помощью Docker Compose. 

В рамках выполнения задания было:

- подготовлено три backend-сервера, работающие на разных портах;
- настроен Nginx как балансировщик нагрузки;
- протестировано распределение запросов между серверами с помощью утилиты pm2;
- добавлены настройки отказоустойчивости через `max_fails` и `fail_timeout`;
- настроен альтернативный пример балансировки через HAProxy.

--

## Практика 23:

Посвящена  контейнеризации, ключевым концепциям Docker, работе в среде WSL (Windows Subsystem for Linux), созданию образов с помощью Dockerfile, управлению многоконтейнерными приложениями через Docker Compose, а также запуску через Docker Compose нескольких экземпляров одного и того же образа приложения на Node.js, описав их как отдельные сервисы.

В рамках выполнения задания были выполнены:

- установка WSL и Docker;
- ознакомление с примером проекта из репозитория — изучение структуру `Dockerfile` и `docker-compose.yml`, запуск примера и протестирован доступный эндпоинт;
- реализация собственного приложения, включающая:
  - три идентичных backend-сервиса на Node.js (Express), каждый из которых отвечает на запрос `GET /` с указанием своего идентификатора (`{"server": "1"}`);
  - Nginx в роли балансировщика нагрузки;
  - `docker-compose.yml`, описывающий все сервисы и объединяющий их в одну сеть;
  - `Dockerfile` для backend-сервиса;
  - папка `nginx` с конфигурационным файлом `nginx.conf` и отдельным `Dockerfile`;
- запуск весь стек командой `docker compose up --build` в WSL;
- проверка балансировки: при повторных запросах `curl http://localhost/` ответ должен был поочерёдно приходить от разных backend-серверов;
- добавление настройки отказоустойчивости через `max_fails` и `fail_timeout` в конфигурации Nginx;
- остановка одного из backend-контейнеров, чтобы убедиться, что Nginx перестаёт направлять на него запросы и продолжает обслуживать трафик через оставшиеся.
